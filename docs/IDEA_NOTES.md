# Project Forge — 아이디어 노트

## 출발점

코딩 에이전트용 stateful memory runtime을 만들려고 시작했다.
에이전트가 세션 간에 경험을 이어가고, 같은 실수를 반복하지 않고, 왜 그런 결정을 했는지 복원할 수 있는 시스템.

## 영감을 준 것들

### MemRL
- 에이전트가 경험에서 학습하는 구조화된 메모리 개념
- failure memory, decision log, write-back 같은 핵심 개념의 기반

### 오케스트레이터 경험
- 여태 만들어본 오케스트레이터들을 기초로 일종의 플랫폼을 만들고 싶었음
- task lifecycle, resume, adapter layer 개념으로 발전

### Karpathy의 Auto Research
- 목표를 주면 알아서 리서치→실행→검증 루프를 도는 개념
- evaluation linkage, next action 추천으로 반영

### AI Hub
- 여러 도구/모델/워크플로우가 모이는 허브 개념
- adapter layer, 다양한 에이전트 연결 구조로 반영

### MiroFish
- 상황 시뮬레이션, 예측 — "이렇게 하면 어떻게 될까"
- v1에서는 패턴 매칭 수준(L1/L2), v2에서 LLM reasoning 기반 예측(L3)

## 핵심 깨달음: 도메인 확장

처음에는 "코딩 에이전트 전용"으로 시작했지만, 본질은 더 넓다.

> **"실수가 비싼 곳에서, 경험이 축적될수록 더 나은 판단을 하게 되는 시스템"**

코딩이든 금융이든 인프라든, 공통점:
- 실패하면 비용이 크다
- 같은 실수를 반복하면 안 된다
- 과거 경험이 다음 의사결정의 질을 좌우한다
- 규칙/제약이 도메인마다 다르다

그래서 **도메인 중립 코어 + 도메인 어댑터** 구조로 전환.
- 코딩 에이전트가 first adapter (직접 쓸 것)
- 금융/리스크 도메인은 second adapter (확장성 증명)
- 토스 같은 금융권에서도 매력적인 시스템이 되면 좋겠음

## 지식 축적 방식

온톨로지처럼 극강의 도메인 정의를 가지고 시작하는 게 아니라:
- **자연 축적**: 써보면서 경험이 지식으로 쌓임
- **초기 시딩**: 원하면 초반에 도메인 지식을 넣어놓을 수도 있음
- 가볍게 시작하되, 쓸수록 똑똑해지는 구조

## 예측 로드맵

| 수준 | 설명 | 시기 |
|------|------|------|
| L1 | 동일 패턴 경고 (exact match) | v1 |
| L2 | 유사 패턴 경고 (vector similarity) | v1 |
| L3 | 조건부 예측 (LLM reasoning over experience) | v2 |
